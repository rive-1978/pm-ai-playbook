## 🗺️ PM workflows

Use these as default sequences. Skip or loop steps as needed based on the decision risk.

| #   | When to use                    | Workflow steps (skills)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 | Outcome                                   | Example                                                                                                                                                                                                                                                                                                                |
| --- | ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Extract and refine job stories | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Define job, context, success signal<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Strip to the root need<br>3. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → List anti-goals and failure triggers<br>4. [Cross-Pollination Engine](../skills/productmanagement-skills/cross-pollination-engine) → Pull 2–3 analogs with proven patterns<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Rank top risks and mitigations           | Job story + success metric + risk list    | **Prompt:** "We have [user type] dropping after week one—what job are they really hiring us for?"<br>**Skill output:** JTBD defines the job + context, First Principles isolates the root need, Inversion lists failure triggers, Cross-Pollination proposes analog patterns, Pre-Mortem ranks the top adoption risks. |
| 2   | Validate a new product concept | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Map motivations + alternatives<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Identify the irreducible value<br>3. [Cross-Pollination Engine](../skills/productmanagement-skills/cross-pollination-engine) → Find adjacent market patterns<br>4. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Define what would kill adoption<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Simulate launch failure paths                      | Concept narrative + must-win assumptions  | **Prompt:** "Stress-test this [product/feature] idea before we build."<br>**Skill output:** JTBD maps motivations + alternatives, First Principles reduces to the core value, Cross-Pollination pulls analog patterns, Inversion lists adoption killers, Pre-Mortem summarizes failure scenarios.                      |
| 3   | Troubleshoot declining metrics | 1. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Break metric into drivers<br>2. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Check if the user job shifted<br>3. [Six Thinking Hats](../skills/testing-skills/six-thinking-hats) → Generate and score hypotheses<br>4. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Identify accelerants of decline<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Test if fixes could backfire                                                     | Prioritized hypotheses + safe experiments | **Prompt:** "[Metric] is down [X%]—find the root cause."<br>**Skill output:** First Principles breaks the metric into drivers, JTBD tests job shifts, Hats ranks hypotheses, Inversion lists accelerants, Pre-Mortem flags fixes with backfire risk.                                                                   |
| 4   | Design complex feature rollout | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Map segments + jobs per segment<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Define minimum viable capability<br>3. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Enumerate segment-specific failure modes<br>4. [Six Thinking Hats](../skills/testing-skills/six-thinking-hats) → Align stakeholder constraints<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Stress-test migration and adoption                             | Rollout plan + migration safeguards       | **Prompt:** "We need a safe rollout plan for [complex feature]."<br>**Skill output:** JTBD maps segment jobs, First Principles defines MVP scope, Inversion lists segment failure modes, Hats align stakeholder constraints, Pre-Mortem highlights migration risks.                                                    |
| 5   | Pivot or persevere decision    | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Revalidate core job and fit<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Separate broken vs intact assumptions<br>3. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Check pivot downside risks<br>4. [Cross-Pollination Engine](../skills/productmanagement-skills/cross-pollination-engine) → Find proven pivot patterns<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Run both-path failure scenarios                        | Decision memo + risk tradeoffs            | **Prompt:** "Should we pivot from [current market] to [new market]?"<br>**Skill output:** JTBD contrasts jobs, First Principles separates intact vs broken assumptions, Inversion lists pivot downside risks, Cross-Pollination pulls pivot patterns, Pre-Mortem compares both-path failures.                          |
| 6   | Optimize user onboarding       | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Define success moment + first job<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Remove non-essential steps<br>3. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Identify immediate churn triggers<br>4. [Cross-Pollination Engine](../skills/productmanagement-skills/cross-pollination-engine) → Borrow proven onboarding patterns<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Validate long-term retention impact           | Shortened onboarding + churn guardrails   | **Prompt:** "Onboarding completion is [X%]—what should we cut?"<br>**Skill output:** JTBD defines success moment, First Principles removes non-essentials, Inversion identifies churn triggers, Cross-Pollination suggests onboarding patterns, Pre-Mortem checks long-term retention risk.                            |
| 7   | Prepare competitive response   | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Understand why customers switch<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Separate differentiation from noise<br>3. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Map response backfire risks<br>4. [Cross-Pollination Engine](../skills/productmanagement-skills/cross-pollination-engine) → Study competitive responses elsewhere<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Test if response creates new liabilities | Response thesis + safe rollout plan       | **Prompt:** "Competitor launched [feature]—how do we respond safely?"<br>**Skill output:** JTBD clarifies switch reasons, First Principles isolates true differentiation, Inversion maps backfire risks, Cross-Pollination pulls response playbooks, Pre-Mortem tests liabilities.                                     |
| 8   | Revitalize a legacy product    | 1. [JTBD Analyzer](../skills/productmanagement-skills/jtbd-analyzer) → Reconfirm original job vs current needs<br>2. [First Principles Decomposer](../skills/productmanagement-skills/first-principles-decomposer) → Separate core value from cruft<br>3. [Inversion Strategist](../skills/productmanagement-skills/inversion-strategist) → Identify modern rejection triggers<br>4. [Cross-Pollination Engine](../skills/productmanagement-skills/cross-pollination-engine) → Pull modernization patterns<br>5. [Pre-Mortem Analyst](../skills/productmanagement-skills/pre-mortem-analyst) → Test power-user backlash                 | Modernization roadmap + safeguard plan    | **Prompt:** "Modernize [legacy product] without breaking power users."<br>**Skill output:** JTBD confirms core job, First Principles separates core from cruft, Inversion lists rejection triggers, Cross-Pollination pulls modernization patterns, Pre-Mortem tests backlash scenarios.                               |

## 📐 Spec-driven frameworks

| Name          | Purpose                                                                                                                     | When to use                                                                                                                                  | Link                                                                       |
| ------------- | --------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| Spec Kit      | Lightweight spec templates and tooling for standardizing feature specs.                                                     | When defining consistent, shareable specs for features and systems.                                                                          | [Github](https://github.com/github/spec-kit)                               |
| OpenSpec      | Open standard for specifying AI agent behaviors, tools, and interfaces.                                                     | When you need structured, interoperable specs for agents and LLM apps.                                                                       | [Github](https://github.com/Fission-AI/OpenSpec)                           |
| BMAD METHOD   | Practical engineering method for scoping, designing, and delivering AI software.                                            | When you want a structured, iterative delivery cadence for AI projects.                                                                      | [Github](https://github.com/bmad-code-org/BMAD-METHOD)                     |
| Get Shit Done | Minimal spec and execution system focused on throughput and accountability.                                                 | When speed and clear commitments are paramount.                                                                                              | [Github](https://github.com/glittercowboy/get-shit-done)                   |
| Design OS     | Design operating system with frameworks and artifacts for product/UX decisions.                                             | When formalizing design decisions and traceability.                                                                                          | [Github](https://github.com/buildermethods/design-os)                      |
| Agent OS 2    | Spec-driven approach and tooling for building production-grade AI agents.                                                   | When standardizing agent capabilities, tools, and evaluations.                                                                               | [Github](https://github.com/buildermethods/agent-os2)                      |
| CC-SDD        | Spec-driven development workflow based on Kiro.                                                                             | When you want a structured, spec-first workflow.                                                                                             | [Github](https://github.com/gotalab/cc-sdd)                                |
| Conductor     | Multi-agent orchestration plugin for Claude Code with parallel execution, evaluation loops, and structured decision-making. | When you need a structured engineering lifecycle — planning, parallel execution, quality gates, and iterative fixes — managed automatically. | [Github](https://github.com/Ibrahim-3d/conductor-orchestrator-superpowers) |
